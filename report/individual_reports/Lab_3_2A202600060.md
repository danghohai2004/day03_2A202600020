- **Log Source**: First experiment run produced no `LLM_METRIC` entries — only `AGENT_START` events followed by immediate errors:
  ```
  {"event": "AGENT_START", "data": {"input": "What is the capital of France?", "model": "gemini-3.1-flash-lite-preview"}}
  ... (no METRIC events)
  Error: 'user_input'
  ```

- **Diagnosis**: After fixing the prompt duplication bug, the `get_system_prompt()` method no longer passed `user_input` to `str.format()`. The `ReAct.v2.txt` template still contained `Query: {user_input}` on line 3, creating a mismatch between the format arguments and the template placeholders. Additionally, the agent was adding the user query separately after loading the template, making `{user_input}` redundant.

- **Solution**: 
  1. Removed the `{user_input}` placeholder from `ReAct.v2.txt` (the query is now appended to the prompt template in the `run()` method).
  2. Added `tools_list` support in `get_system_prompt()` for backwards compatibility with `ReAct.v1.txt` which uses `{tools_list}` on line 17.

## III. Personal Insights: Chatbot vs ReAct (10 Points)

1. **Reasoning**: The `Thought` block forces the LLM to articulate its reasoning before acting, which reduces hallucination. For example, on Q10 (Mount Everest vs Eiffel Tower height), the agent explicitly searched both values separately, then calculated the difference — a multi-step process the LLM wouldn't reliably do without the structured prompt. The Thought block acts as a forcing function for decomposition. This value amplifies with structured tool returns: when a tool returns `{"valid": False}`, the agent can explicitly reason over why that matters for the overall query, rather than parsing through a verbose error string.

2. **Reliability**: The Agent performed *worse* on simple queries. On question '234 * 56', the chatbot answered directly with the correct result in ~140 tokens. The agent used calculator + verification in ~1,202 tokens — 9x the token cost for the same answer. For well-known facts and basic math, the tool overhead is wasteful. Additionally, some math questions (Q9, Q15) were answered correctly by the agent *without* using tools, showing that even with a strong prompt, the model sometimes skips the ReAct loop when it's confident — which means the answers are correct but not verifiable through the system.

3. **Observation**: Tool observations provide concrete feedback that shapes the next step. On question 'Japan population * 2', the agent first searched for the population, received "approximately 123.8 million," then used the calculator to multiply. The observation grounded the calculation in real data rather than the LLM's training data estimate. The new e-commerce tools return named fields (`item`, `quantity`, `available`, `weight_kg`, `shipping_cost`, `coupon_code`, `discount`, `valid`) that provide richer ground truth than plain string observations. The LLM can reason over `valid: False` as a branching signal, not just a text string to absorb. However, when web search returned empty results (tool failure), the agent correctly fell back to using internal knowledge — showing graceful degradation.

4. **Compositional Power of Heterogeneous Tools**: The ReAct agent's true strength emerges not from using individual tools, but from orchestrating multiple heterogeneous tools in a single query. `check_stock` returns `{quantity, available}` which can gate whether the agent proceeds to `calc_shipping` or `get_discount`. A Chatbot cannot condition its behavior on the output of a stock check — it has no execution path. When `available: False`, the agent can branch to suggest alternatives; the chatbot has no mechanism to receive or react to that signal.

## IV. Future Improvements (5 Points)

- **Scalability**: Use an asynchronous queue for tool calls so multiple tools could execute in parallel when the agent identifies independent information needs. In the e-commerce flow "order MacBook to Hanoi with VIP coupon", `check_stock`, `get_discount`, and `calc_shipping` are independent and could execute in parallel. Only the final calculator step depends on their outputs. Parallel tool execution would reduce end-to-end latency.

- **Safety**: Implement a supervisor LLM that audits the agent's proposed actions before execution — especially important for tools like `web_search` that could be directed at malicious URLs or used for prompt injection. The calculator's `eval()` should be replaced with `ast.literal_eval` or a dedicated math expression parser.

- **Performance**: Add a tool retrieval step for many-tool systems. Instead of injecting all tool descriptions into every prompt, use embedding-based retrieval to only show relevant tools, reducing prompt size and improving accuracy.

- **Rate Limiting**: Implement exponential backoff and retry logic in the agent's `llm.generate()` call path. The free-tier 429 errors caused 13% failure rate on agent v1 — a production system cannot fail silently on quota exhaustion. A circuit breaker pattern would prevent cascading failures.

- **Consistent Tool Output Format**: New tools return Python dicts that get stringified via `str()` in `agent.py` line 87, producing inconsistent boolean formatting (`True` vs `true`) and relying on dict insertion order. Serializing to `json.dumps` would produce consistent LLM-friendly representations aligned with structured function-calling patterns.

- **Structured Output**: Move from regex-based parsing to structured JSON output (function calling / tool use API). This eliminates the "could not parse Action" error mode entirely and provides guaranteed format compliance.
